앞서 언급한 모바일 기반 서비스를 제공하기 위해 필요한 보안 아키텍처를 설계하거나 보안 솔루션을 적용할 때 근간이 되는 보안 요소 기술에 대한 이해가 필요하게 됩니다. 이에 대한 체계적인 설명은 보안 이론 개론서를 참고하면 좋을 것 같습니다. 본 장에서는 실무적으로 보안 아키텍처 설계 접근법에 대해 언급하고 보안요소 기술에 대하여 개략적인 설명을 하는 것을 목표로 합니다.

보안 아키텍처를 설계할 때 자주 참조 모델로 언급되는 것이 ITU-T X.805 표준입니다. 이는 잠재적 보안 위협과 이에 대응하여 시스템 계층(클라이언트, 네트워크, 서버, 응용, 데이터) 별로 정보보호 기술영역과 적용기술을 정의합니다. 이러한 구조는 범용적으로 적용 가능한 참조 모델로서 유사한 신규 서비스(IoT, 5G 등)에도 적용이 가능합니다.

이어지는 글에서는 보안 아키텍처의 설명으로부터 세부 보안기술 구성요소에 대한 설명을 할 것입니다.

1 - 보안 아키텍처(ITU-T X.805)

2002년 벨 연구소(Bell Labs)의 연구원들은 단순한 의문을 가지게 되었습니다.

“차 세대 네트워크의 보안을 위해 어떤 표준을 사용해야 할까?” 이어서 두 번째 질문을 하게 되었습니다. “어느 정도의 보안이 충분하고, 보안성은 어떻게 측정해야 할까?” 이에 대한 문헌과 자료를 찾아보았지만 충분하고 적합한 답을 구할 수 없게 되자 그들은 직접 보안 프레임워크를 만들게 되었고, ITU-T 기관의 보안 표준1으로 채택되기에 이르렀습니다.

표준의 서문은 다음과 같습니다. 서문의 내용은 보안 시스템 자체가 아닌 네트워크 보안 설계에 초점이 맞춰져 있는 것을 알 수 있습니다.

The telecommunications and information technology industries are seeking cost-effective comprehensive security solutions. A secure network should be protected against malicious and inadvertent attacks and should have high availability, appropriate response time, reliability, integrity, scalability, and provide accurate billing information. Security capabilities in products are crucial to the overall network security (including applications and services). However, as more products are combined to provide total solutions, the interoperability, or the lack thereof, will define the success of the solution. Security must not only be a thread of concern for each product or service but must be developed in a manner that promotes the interweaving of security capabilities in the overall end-to-end security solution. To achieve such a solution in a multi-vendor environment, network security should be designed around a standard security architecture.

이 표준에서는 시스템의 보호대상 식별 방법으로 계층 별 분류 방법으로 3계층(인프라, 서비스, 응용)으로 분류하거나 행위 별 분류 방법으로 3가지(관리, 제어, 사용자)로 보안 프레임워크를 구분합니다.

식별된 보호대상에 대하여는 8가지의 보안 단위기술(Security Dimensions)를 적용하여 보안위협2및 취약성 분석을 하고, 정보보호 목표 또는 요구사항을 도출하게 됩니다.

8가지 보안단위기술은 다음과 같습니다.

단위기술 내용
1. 접근 제어/권한 통제(Access Control) 허가 받지 않은 사용자, 장비, 네트워크/정보시스템/서비스에 대한 접근을 제한합니다.
2. 인증(Authentication) 정보자산에 접근하고자 하는 사용자와 장비에 대해 정보 자산의 중요도에 따라 차별화된 식별성을 보장합니다.
3. 부인방지(Non-repudiation) 사용자 및 장비의 정보자산에 대한 접근/처리 내역을 보존하여 감사/추적성 및 모니터링 증적을 확보합니다.
4. 데이터 기밀성(Data Confidentiality) 정보의 허가 받지 않은 유출 로부터 보호합니다.
5. 통신보안(Communication) 정보가 인가된 종단 사이에서만 통신이 되도록 보장합니다.
6. 데이터 무결성(Data Integrity) 정보가 인가받지 않은 자에 의해 수정, 삭제, 복제되지 않도록 합니다.
7. 가용성(Availability) 인가된 사용자가 정보시스템 및 정보 자원을 필요로 할 때 접근하고 사용할 수 있도록 보장합니다.
8. 개인정보보호(Privacy) 개인정보의 유출 및 오, 남용 방지를 위해 주요 경로에 대한 보호대책을 적용합니다.

상기 보안 참조 모델에 따라 KISA(한국인터넷진흥원)에서 제시하는 보안 아키텍처 모델 은 다음과 같습니다.

보안 계층(Security Layer)는 5단계로 구분하였는데, 인프라시스템(서버, 네트워크, 시설), 어플리케이션, 데이터(DB), 단말기(Client), 인터페이스로 구분하였습니다. 상기 8개의 보안단위기술은 6가지(인증, 접근통제, 데이터보호, 암호화, 감사 추적, 위협 탐지)의 보안요소로 적용합니다. 이와 더불어 영역별 보안기능을 정리하여 서비스 형태 별로 정보자산의 특성에 따라 아키텍처 모델을 설계할 수 있도록 합니다.

이 때 모델에 의해 도출된 보안 목표 기능(Security Objectives)는 목표 시스템의 정보보안을 구현하기 위한 통제대책(Security Measures)으로 침입탐지, 침입차단, 백신, DB암호화, DB접근통제 등을 의미하는 것으로서 정보보호 솔루션으로 보안기능을 구현할 수 있습니다.

예를 들면, 데이터 보안 계층(Security Layer)에서 6가지 보안 요소 순으로 보안 기능을 도출해 보자. 인증(ID/PW, DB접근통제), 접근통제(DB접근통제), 데이터 보호(스팸메일 차단, 바이러스 침입차단), 암호화(암호화 저장), 감사(DB접속 로그관리), 위협탐지(정보유출방지)으로 도출이 가능합니다.

이와 같이 보안 계층 별, 보안 요소를 적용하여 각각 취약점 분석 및 보안 위협에 따라 정보보호 기술 요구항목을 도출할 수 있습니다.

상기 보안요소(Security Elements)는 암호 기술을 기반으로 한 인증, 접근 및 권한 통제, 부인 방지, 개인정보보호 등의 기술로서 하기에 보다 상세히 기술하고자 합니다.

2 - 암호 기술

정보보안 분야에서 암호는 모든 것의 알파와 오메가로 생각하면 될 것입니다.

보안 솔루션은 기본적으로 암호 기술을 기반으로 하여 정보를 보호한다고 볼 수 있습니다. 하지만 암호에 대한 일반적인 인식은 어렵다는 것입니다. 왜냐하면 암호 이론이 매우 어렵기도 하고 수학적 배경 지식을 이해하고 있어야 하기 때문입니다. 그러나 암호를 적용하여 개발하는 데 있어 이론을 전부 이해할 필요는 없다고 생각합니다. 대략적인 개념을 파악하고 어떤 암호 모듈의 암호 알고리즘을 선택하여 적용할 수 있다면 실무적인 관점에서는 문제가 없기 때문입니다.

암호는 기본적으로 중요 정보가 처리되는 과정에서는 대부분 필요한 것이지만 성능상의 문제가 있기 때문에 주로 인증 과정에서 적용을 하게 됩니다. 이러한 선택 기준에서 법규를 참고할 필요가 있습니다. 왜냐하면 법규에서는 필수적인 적용 범위를 설정해 두기 때문입니다. 하지만 이러한 규정은 최소 범위라는 점을 유념할 필요가 있습니다. 즉, 서비스를 제공하는 입장에서는 고객의 정보를 보호하기 위해 범위에 얽매일 필요 없이 최선을 다해야 하기 때문입니다.

이하 절에서는 암호 적용을 규정하는 법규와 암호 기술을 알아보기로 합니다. 특히 금융권을 중심으로 알아보기로 하는데, 금융 쪽이 보안의 최전선에 있으며, 규정의 잣대가 상대적으로 높기 때문에 이를 감안하여 어느 정도 수준에서 보안을 적용해야 할지 알 수 있기 때문입니다. 본문의 내용은 금융보안연구원의 암호 기술 가이드라인 을 참고로 작성하였습니다. 1

2.1 - 암호화 대상

정보보안 분야에서 암호는 모든 것의 알파와 오메가로 생각하면 될 것입니다.

법규에서 규정하는 암호화 대상

금융권에서 암호화 적용대상은 크게 내부에 저장하는 금융 관련 저장 정보와 전자금융 거래 시 전송되는 전송 정보로 구분할 수 있습니다.

전자금융 감독규정은 전자금융업의 안전성 기준 강화를 위한 내용을 중점적으로 다루고 있습니다. 하기 표와 같이 네트워크 상에서의 보안, 특히 무선통신망에서의 불법 접속 방지, 암호 기술 적용 시 키 관리 방안 마련, OTP 인증, 보안 프로그램(백신) 적용, 무결성 검증, 공인인증서 적용 등이 주요 내용이 됩니다. 공통적으로 언급되는 것은 암호화 적용이 언급된다는 점입니다.

제31조 (암호프로그램 및 키 관리 통제)

전자자금 이체 시 보안카드를 포함한 일회용 비밀번호를 적용, 암호화 통신, 이용자 전자적 장치에 보안 프로그램 설치 등 보안대책 적용, 전자금융거래프로그램의 위·변조 여부 등 무결성 검증 방법 제공 등

제34조 (전자금융거래 시 준수사항)

전자서명법에 의한 공인인증서 또는 이와 동등한 수준의 안전성이 인정되는 인증방법 사용

제37조 (공인인증서 사용 기준)

전자금융감독 규정 시행령

이용자 인증, 정보처리시스템 인증, 통신채널 암호화, 전자금융 거래내역의 무결성, 전자금융 거래내역의 부인방지

제7조 (인증방법의 안전성평가 등)

이 외에도 금융회사는 저장해야 하는 정보에 대해, 필요시 개인정보보호법,정보통신망법, 신용정보업 감독규정 등의 법규를 준수하여 개인식별정보 및 주요 금융정보 등을 암호화 전송 또는 저장해야 합니다. 전자금융 관련법 이외의 법규에서 암호화 관련한 주요 내용은 표 2에 기술하였습니다.

개인정보보호법은 여타의 법규에서도 유사한 내용을 담고 있는 것을 알 수 있는데, 개인정보에 관련해서는 기본법으로 생각하면 될 것입니다. 최근에는 개정이 되어 정보통신망법에서 중복되는 규정이 개인정보보호법으로 일원화되기도 했습니다. 또한 국제적인 유럽연합의 GDPR 법령을 수용하려는 움직임이 일고 있어 한층 개인정보 보호 권리를 강화하는 추세에 있는 점을 명심할 필요가 있습니다.

법규의 내용 중 개인식별정보를 처리하는 과정에서 암호화 등 안전성 확보에 필요한 조치를 취할 것을 규정하고 있습니다.

개인정보보호법

개인정보처리자가 고유식별정보를 처리하는 경우 고유식별정보가 분실·도난·유출·변조 또는 훼손되지 아니하도록 암호화 등 안전성 확보에 필요한 조치를 할 것

제24조 (고유식별정보 처리의 제한)

개인정보처리자는 개인정보가 분실, 도난, 유출, 변조 또는 훼손되지 아니하도록 내부관리계획 수립, 접속기록 보관 등 전성 확보에 필요한 기술적·관리적 및 물리적 조치를 할 것

제29조 (안전조치의무)

신용정보업 감독규정

신용정보 회사 등은 개인신용정보의 암호화 등 기술적·물리적·관리적 보안대책의 구체적인 기준 마련

제20조 (기술적·물리적·관리적 보안대책)

정보통신망법, 개인정보보호법에서 개인정보 등이 분실, 도안, 유출 또는 훼손되지 아니하도록 암호화를 하도록 규정되어 있는데, 여기서 암호화가 필요한 정보는 두 가지로 크게 구분할 수 있습니다.

첫번째는 암호화된 정보를 다시 복호화 할 수 없는 정보입니다. 즉, 비밀번호가 이에 해당합니다. 정보주체를 제외하고 정보를 다루는 관리자조차 암호화된 정보의 원래 정보가 무언인지를 알 수 없어야 하는 정보가 됩니다.

두번째는 암호화된 정보를 다시 복호화 할 수 있어야 하는 정보입니다. 즉, 바이오 정보, 주민등록번호, 신용카드번호, 계좌번호 등이 이에 해당합니다. 정보 저장 시에는 암호화되어 있어야 하나 사용 시에는 복호화가 가능하여 원래 정보를 알 수 있어야 하는 정보입니다. 이러한 암호화 대상 정보는 데이터베이스에 저장하는 경우는 물론이고, 정보통신망을 통해 송, 수신되는 경우에도 반드시 암호화되어야 합니다.

암호기술 적용 예

암호 기술을 적용하는 사용예를 대표적으로 나열하면 다음과 같습니다. 온라인 상에서 사용자의 입력 정보 (키보드 보안 등), 거래 시 추가 인증(OTP, 인증서 등), 전송 정보 보호(E2E 암호화) 등이 있으며, 오프라인에서 IC 카드(카드 인증, 거래 정보 확인 메커니즘), VPN, 통신 암호화, DB 보안 등에서 암호 기술이 적용되어 사용됩니다.

전자서명 및 인증

  • 전자 서명

금융권에서는 금융 거래에서 신원 인증 및 부인 방지의 목적으로 전자서명 기술을 이용합니다. 전자서명은 뒤에 자세히 다루겠지만, 원본 전문의 해시 값을 계산하여 전자서명을 하고 상대 측에서 원본의 진위 여부를 확인하는 방식입니다. 전자 서명된 문서는 서명(Signing) 시 사용한 공개 키는 인증 기관에서 등록, 관리하며 누구든 공개키에 접근이 가능하고 서명 값 검증 시에 사용하게 됩니다. 이러한 공개적 접근을 위해서는 공개 키 인증서(Certificate)와 인증기관(CA, Certificate Authorities), 등록기관(RA, Registration Authority), 저장소(Repository) 등이 필요한데, 이러한 전자 서명된 인증서 분배를 통해 공중망에서 상호인증이 가능한 시스템을 공개 키 기반 구조(PKI, Public-Key Infrastructure)라고 합니다. 이러한 암호 기술을 이용한 전자 서명은 온라인 뱅킹, 온라인 결제, IC 카드 인증 등의 다양한 분야에서 이용합니다.

  • 인증

금융권에서 많이 쓰이는 인증 수단으로 OTP(One Time Password) 발생기를 들 수 있습니다. 인터넷, 스마트폰 등의 단말로 인터넷 뱅킹 등의 서비스를 이용할 때 사용하며, 매번 바뀌는 일회용 생성 비밀번호이므로 기존 비밀번호 인증 방식 보다는 안전하다고 할 수 있습니다. 암호 기술로는 해시 함수, 블록 암호, 난수 발생기를 사용하여 일회용 비밀번호를 생성합니다.

입력정보 암호화

  • E2E 기반 가상 키보드

PC 및 스마트폰 단말을 이용하여 금융 서비스를 이용할 때 특히, 비밀번호, 계좌번호 등의 중요 정보를 입력할 때 정보가 해킹에 의해 노출될 위협이 존재합니다. 이 때 입력 값을 보호하기 위해 가상 키보드 또는 키보드 암호화 기술을 사용하는데, 메모리 해킹 사고 등을 방지하는 효과적인 수단이 됩니다. 현재 금융권에서는 E2E암호화, 가상 키 패드 등을 의무적으로 적용하도록 권고하고 있습니다.

인증서 비밀번호, 계좌 비밀번호, 보안카드 번호 등의 중요 정보 입력 시 입력된 값을 암호화하며, 사용자에게는 이와 연동된 커스텀 인터페이스(가상 키보드)로 표출되어 운영체제에서 제공하는 기본 키보드와 다른 방식으로 입력하게 됩니다. 암호화된 입력 값은 서버까지 전자서명 및 통신 구간에서 암호화되어 전송되어 E2E 암호화가 동시에 달성되는 구조입니다.
